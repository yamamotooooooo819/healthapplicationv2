## 1. プロジェクト概要

食事内容と PFC バランス(たんぱく質・脂質・炭水化物)を記録・可視化する Web アプリ。
実運用(第三者に公開)を前提とする。

### 技術スタック(確定)
- フロントエンド: React + **TypeScript**、ビルドツールは **Vite**
- バックエンド: Java / Spring Boot、ビルドツールは **Maven**
- データベース: PostgreSQL(16 で検証済み)
- DBスキーマ管理: **Flyway**(検証済み `schema.sql` を `V1__initial_schema.sql` として使う)
- 認証: **Firebase Authentication** に委譲(Spring Boot 側でトークンを検証)
- ホスティング: **保留**。開発中はローカルPCで進め、公開直前に決定する。
  第一候補はフロント= Vercel または Netlify、バック+DB= Railway(スリープ回避のため)。
- 画像解析 AI: 第3段階で Python / FastAPI の独立サービスとして構築(本書の対象外)

### アーキテクチャ方針
- フロント / バックエンド / DB を分離する。
- 設定値(DB接続情報・APIキー等)はコードに直書きせず**環境変数**で管理する。
  無料枠 → 自宅サーバー → AWS への移行を見据えるため。

## 3. 確定した設計判断(変更禁止・要確認)

壁打ちで決定済み。これらは実装の前提であり、勝手に変えないこと。

1. **認証は外部サービスに委譲する。** `users` テーブルにパスワードは保持しない。
   外部サービスが発行する ID を `auth_provider_id` に保存して紐づける。
2. **主キーは `users` のみ UUID。** ID 推測攻撃を防ぐため。他テーブルは `BIGINT GENERATED ALWAYS AS IDENTITY`。
3. **栄養素は `NUMERIC` 型を使う。** `float`/`double` は誤差が出るため禁止。
4. **栄養項目は PFC + カロリーのみ**(MVP)。食物繊維・塩分・ビタミン等は持たない。
5. **食事区分は 4 種類**: `breakfast` / `lunch` / `dinner` / `snack`。`CHECK` 制約で縛る。
6. **食事記録はスナップショット方式。** 食べた時点の PFC 実数を `meal_entry_items` にコピー保存し、
   後から食品マスタが変わっても過去の記録は不変にする。
7. **同一食事内の同一食品の重複は許可する**(おかわり等を表現するため制約をかけない)。
8. **My食事テンプレートはスナップショットしない。** 構成(food_id + quantity_g)のみ保持し、
   常に最新の食品マスタを参照する。食事記録へ「展開」する瞬間に PFC を計算してスナップショットする。
9. **削除は原則として論理削除**(`deleted_at` カラム)。物理削除はしない。
   集計・履歴に影響するため。
10. **栄養値はすべて「可食部100gあたり」で `foods` に保存**し、`quantity_g` で都度換算する。

### 技術選定の根拠(勝手に乗り換えないこと)
- **ビルドツールは Maven**: 情報量と安定性を優先。Gradle ではない。
- **DB管理は Flyway**: 検証済み `schema.sql` をほぼそのまま活かせるため。Liquibase の多機能さは PostgreSQL 一本の今回には不要。直流し(ツールなし)は実運用で履歴が残らず禁止。
- **フロントは TypeScript + Vite**: 型安全で API 境界のバグを減らし、AI による実装精度も上がる。Next.js は自身がバックエンド機能を持ち、Spring Boot と役割が衝突するため不採用。CRA は非推奨のため不採用。
- **認証は Firebase Authentication**: Spring Boot を主役にする構成のため、「認証だけ外部、ロジックとDBは Spring Boot」という分離が最も素直。Supabase は認証+DB+API を一体提供し Spring Boot の役割と重複するため、今回は不採用(Spring Boot の比重を下げたい方針に変わった場合のみ再検討)。

## 4. データベース設計

スキーマは `schema.sql` に定義済み(PostgreSQL 16 で作成・制約動作を検証済み)。
**`schema.sql` を正とする。** エンティティはこのスキーマに一致させること。

### テーブル一覧
| テーブル | 段階 | 役割 |
|---|---|---|
| `users` | 1 | 利用者。認証情報は外部サービスが管理 |
| `foods` | 1 | 食品マスタ(100gあたりのPFC・kcal) |
| `meal_entries` | 1 | ある日・ある区分の食事のまとまり |
| `meal_entry_items` | 1 | 食事の明細。PFCをスナップショット保存 |
| `my_meals` | 2 | My食事テンプレート(名前のみ) |
| `my_meal_items` | 2 | My食事の構成(スナップショットなし) |
| `favorites` | 2 | お気に入り食品。(user_id, food_id) で一意 |

### 重要な関係
- `meal_entries (1) --- (N) meal_entry_items` … 親削除時 `ON DELETE CASCADE`
- `foods (1) --- (N) meal_entry_items` … 食品は参照される側
- `my_meals (1) --- (N) my_meal_items` … 親削除時 `ON DELETE CASCADE`

## 6. 文科省データ取り込みの注意点

初期食品データは文部科学省「日本食品標準成分表」(Excel等で公開)を取り込む。
これはリアルタイム API ではなく、**公開ファイルをダウンロードして自前 DB に投入する**。

取り込み時のデータクレンジング規則(数値以外の記号が混在する):
- `Tr`(微量)→ `0` として扱う
- `-`(未測定)→ `NULL` として扱う
- `(数値)`(推定値、括弧付き)→ 括弧を外して数値として扱う
- 取り込んだ食品は `source = 'mext'` で登録する

**ライセンス**: 文科省データは出典明記等の条件があるため、公開前に最新の利用条件を一次情報で確認すること。

**補足**: 文科省データは素材中心(精白米、鶏むね肉など)で、市販の加工食品は含まれない。
将来 Open Food Facts(`source='openfoodfacts'`)等で補完する余地を残してある。

## 7. 実装ルール(Claude Code への指示)

- **一度に全部作らない。** §5 のチェックリストを上から1つずつ、小さく実装して確認する。
- 各タスク着手前に、関連する「設計判断(§3)」を確認する。
- スナップショット計算は必ずサーバー側(Spring Boot)で行う。フロントの計算値を信用しない。
- 機密情報(接続文字列・APIキー・トークン)をコードやログに出力しない。
- 健康データを扱う以上、画像解析の推定値には「これは推定値です」と明示する設計を第3段階で入れる。
- PFC 集計は件数が増えるとフロント計算が重くなるため、サーバー側の集計 API で返す。
